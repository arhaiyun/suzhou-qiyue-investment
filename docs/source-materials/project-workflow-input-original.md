# 创业项目工作流输入清单
## 一、项目初步定位
本项目拟开发一款“实体陪伴型智能挂件 + 云端数字生命体”的软硬件结合产品。
产品实体形态初步可以理解为一种可随身携带的毛绒类互动挂件，例如包挂、尾巴、耳朵、毛球、异形小生物等。其核心不是传统意义上的玩偶，而是具备触觉感知、动作反馈、温度反馈和云端成长记录能力的“随身情绪陪伴硬件”。
云端部分则是实体产品背后的完整数字生命体。实体挂件可能只是这个数字生命体的一部分，例如一条尾巴、一只耳朵、一个身体部件；而在手机、电脑或小程序端，用户可以看到完整的 2D 虚拟形象，例如小狐狸、小猫、小兔子，或其他更具记忆点的原创角色。用户通过日常触摸、携带、捏揉、抚摸、互动等行为，逐步培养它的性格、故事和关系记忆。
- 项目核心可以概括为：
一个可以被摸、会反应、能陪伴、会成长，并在云端形成个人故事的智能情绪挂件。

## 二、产品形象要求
### 1. 外观方向
产品实体外观不应被限定为某一种具体动物或造型。当前设想中有“小狐狸尾巴毛绒形态”这一创意，但这只是候选方向之一。
- 后续可由模型继续发散更多形象创意，包括但不限于：
- 小狐狸尾巴
- 小猫尾巴
- 小兔耳朵
- 毛绒小怪物
- 异形生物器官
- 情绪毛球
- 赛博宠物挂件
- 不完整但有生命感的身体部件
### 2. 形象原则
外观不一定要走传统“可爱路线”，可以有一点怪、有一点丑、有一点奇特，但必须有强记忆点。
- 核心要求是：
一眼能记住；
有陪伴感；
有情绪表达空间；
适合被随身携带；
适合被触摸、抚摸、揉捏；
能够承载后续 IP 化和数字生命体设定；
实体形象与云端完整 2D 形象之间存在合理关系。
### 3. 实体与数字形象关系
实体产品可以是数字生命体的“现实身体部件”或“现实化身”。
- 例如：
用户手里的实体挂件是一条尾巴，而手机里的完整形象是一只完整的小狐狸。用户每天与尾巴互动，云端的小狐狸会根据这些互动数据产生情绪、记忆、故事和成长变化。
- 这种关系可以形成一种独特的产品设定：
实体是入口，数字生命体是完整人格；触摸是数据，故事是反馈。

## 五、实体产品核心功能
- 功能一：用力捏会“炸毛”
用户用力捏实体挂件时，产品会通过柔性传感器识别触感、力度和触摸区域，并作出“炸毛”反应。
- 这一功能的体验重点是：
有明显情绪反馈；
像小动物受到惊吓或被激怒；
能体现“它不是死物，而是有反应的生命体”；
反应可以是毛发蓬起、震动、轻微动作、声音提示或灯光反馈。
- 技术设想：
在毛绒表层或关键触摸区域配置柔性传感器；
识别用户捏压强度和频率；
由小型微处理器判断触发条件；
通过机械结构、微型马达、震动模块或材料变化实现炸毛效果。
- 功能二：被抚摸会扭动
用户轻轻抚摸时，产品会出现类似真实尾巴或小动物身体的扭动反应。
- 这一功能的体验重点是：
轻柔；
有生命感；
像被安抚、撒娇或回应；
与“用力捏会炸毛”形成情绪反差。
- 技术设想：
柔性传感器识别抚摸方向、速度、频率；
微处理器判断是“抚摸”而非“捏压”；
小型驱动结构让尾巴或挂件产生摆动、扭动、轻颤等动作。
- 功能三：可发热
产品可以发热，在冬天作为暖手或暖脖子的工具。
这一功能既有实用价值，也能增强“它有体温”的陪伴感。
- 体验重点：
温暖但安全；
温度可控；
像小动物体温；
可与情绪状态联动，例如开心时变暖、安慰用户时升温。
- 技术设想：
内置温控模块；
配置温度传感器；
设置安全温度上限；
支持不同温度档位；
未来可与软件情绪状态联动。
- 功能四：连接云端程序，形成个人故事流
实体产品可以与云端程序或终端软件连接，不定期同步更细致的互动数据。
- 同步数据包括但不限于：
用户触摸次数；
抚摸频率；
捏压强度；
携带时间；
使用时段；
发热使用次数；
互动偏好；
情绪推测；
用户输入的文字、语音或日常故事。
这些数据会在云端转化为属于用户自己的“数字人日志”或“角色成长故事流”。
- 例如：
今天它被你摸了很多次，所以它觉得你可能有点累；
今天它一直待在你的包上，陪你去了三个地方；
它记住了你晚上最常和它说话；
它为你生成一段只有你们两个知道的小故事；
它的性格会随着你的互动方式逐渐变化。

## 六、云端软件 / 小程序功能设想
### 1. 2D 数字形象展示
云端软件中呈现该实体挂件背后的完整 2D 数字形象。
实体可能是一条尾巴，云端中则是一只完整的小狐狸；实体可能是一只耳朵，云端中则是一只完整的小猫、小兔或原创生物。
- 2D 数字形象可支持：
基础动作；
表情变化；
情绪状态；
服装更换；
居住空间；
关系等级；
记忆记录；
用户定制。
### 2. 互动机制
云端角色可通过规则引擎或大模型实现线上互动。
- 早期版本可采用规则引擎实现基础互动，例如：
每天问候；
根据触摸次数生成反馈；
根据时间段触发不同对话；
根据用户行为生成简单故事；
根据互动频率改变亲密度。
- 进阶版本可接入大模型，实现更智能化的陪伴能力，例如：
自然语言对话；
个性化回应；
记忆用户偏好；
生成每日故事；
形成角色人格；
结合实体互动数据生成专属叙事；
根据用户心情提供安慰、陪伴、鼓励或调侃。
### 3. 数字人日志 / 故事流
软件端应形成一个核心内容模块，即“属于用户自己的故事流”。
故事流不是普通打卡记录，而是把用户与实体挂件之间的互动转化为一个持续发生的关系叙事。
- 例如：
今日互动日记；
本周陪伴故事；
角色心情变化；
用户与角色的共同记忆；
根据真实互动生成的虚构小剧场；
角色写给用户的信；
用户理想生活或幻想故事在 2D 形象中展现。
### 4. 虚拟商城
软件端可设置虚拟商城，用于后续商业化扩展。
- 可售卖内容包括：
2D 角色服装；
虚拟房子；
家具摆件；
情绪道具；
节日皮肤；
特殊故事包；
限定角色形象；
互动动作；
关系纪念品。
这里存在“硬件一次性销售 + 软件持续付费 + 虚拟内容消费”的复合商业模式可能。

## 七、智能性要求
### 1. 实体硬件智能性
实体包挂产品不需要一开始实现很复杂的大模型能力，核心是实现稳定、可感知、可反馈、可记录的本地智能。
- 硬件端初步智能性包括：
柔性传感器识别触摸；
温度传感器识别发热状态；
小型微处理器处理基础交互逻辑；
本地短期存储互动数据；
支持蓝牙或其他方式与终端同步数据；
可触发简单动作反馈；
可触发温度反馈；
可区分捏压、抚摸、长按、频繁互动等行为。
实体硬件的智能目标不是“会聊天”，而是“能感知、能反馈、能留下行为数据”。
### 2. 云端软件智能性
软件端是更高阶智能的主要承载区。
- 软件端智能性包括：
通过规则引擎实现基础养成逻辑；
通过大模型实现自然语言互动；
根据硬件交互数据生成角色反应；
根据长期数据形成用户画像；
根据用户偏好改变角色性格；
生成日常故事、日记、信件、互动小剧场；
支持用户自定义角色设定；
支持虚拟空间和商城系统；
形成长期陪伴关系。
### 3. 智能性阶段划分
- 建议将智能性分为三个阶段：
第一阶段：规则驱动型互动
重点是硬件能动、能热、能记录，软件能展示角色、亲密度和简单故事。
第二阶段：轻量大模型陪伴
重点是角色能聊天、能记忆、能根据互动数据生成每日故事。
第三阶段：人格化数字生命体
重点是角色有长期记忆、稳定人格、成长线、用户关系史和个性化世界观。

## 九、Demo 方式初步设想
### 1. Demo 核心目标
Demo 不需要一开始展示完整商业化产品，而应优先证明以下几点：
这个东西有生命感；
用户摸它，它会有反应；
用户用不同方式互动，它会有不同反馈；
实体互动数据可以同步到软件端；
软件端角色能根据这些数据生成故事；
用户能感受到“我和它之间正在建立关系”。
### 2. Demo 版本一：硬件交互 Demo
展示一个实体毛绒尾巴或毛绒挂件原型。
- 需要演示：
用力捏后炸毛或震动；
轻轻抚摸后扭动；
开启发热后有体温；
设备能记录基础互动次数；
设备能通过蓝牙或数据线把互动数据同步到终端。
### 3. Demo 版本二：软件故事 Demo
展示一个手机端或网页端 2D 角色界面。
- 需要演示：
角色形象展示；
今日互动数据同步；
根据用户抚摸、捏压、发热使用情况生成一段日记；
角色对用户说话；
角色心情变化；
简单亲密度或成长值系统。
### 4. Demo 版本三：完整体验 Demo
把硬件与软件串联起来。
- 演示流程可以是：
用户摸了实体尾巴 → 尾巴扭动 → 数据同步到手机 → 手机里的小狐狸出现反应 → 小狐狸生成一句话或一段故事 → 用户感受到实体与数字角色之间存在真实关系。
### 5. Demo 最关键体验
Demo 的关键不是技术复杂度，而是“情绪闭环”。
- 完整闭环应为：
触摸实体 → 实体反馈 → 数据记录 → 云端理解 → 角色表达 → 用户产生情感连接。

## 十一、需要朋友进一步用模型分析的问题
### 1. 产品定义问题
- 这个产品最准确的品类应该是什么？
是 AI 玩具、智能包挂、情绪陪伴硬件、虚拟宠物入口，还是软硬件 IP 产品？
第一代产品应主打“好玩”“陪伴”“情绪价值”“暖手实用”还是“AI 互动”？
- 实体与云端之间的主次关系如何设计？
### 2. 形象创意问题
- 除小狐狸尾巴外，还有哪些更有记忆点的形象？
是否应该走可爱、丑萌、怪诞、潮酷、轻成人、二次元、赛博宠物等不同方向？
- 哪种形象更适合年轻人主动挂在包上？
- 哪种形象更适合未来 IP 化？
- 形象如何避免与普通毛绒玩具同质化？
### 3. 技术可行性问题
- 柔性传感器布置在毛绒产品上是否可行？
- “炸毛”效果用什么技术路径实现最稳妥？
- 尾巴扭动用什么机械结构最合适？
- 发热模块如何保障安全？
- 电池容量、重量、续航、散热、安全认证如何平衡？
- 第一代 Demo 应做到什么技术程度即可打动投资人或用户？
### 4. Demo SOP 问题
- Demo 应分几个版本？
- 每个版本需要展示哪些功能？
哪些功能必须真实实现，哪些可以用 Wizard of Oz 或半模拟方式展示？
Demo 面向投资人、用户、供应链、IP 方时，重点是否不同？
- 如何用最低成本验证用户是否真的愿意买？
### 5. 商业模式问题
299—499 元和 899—999 元两个价格区间是否合理？
- 国内年轻人是否愿意为这类产品付费？
- 海外成人市场是否存在更高定价空间？
- 硬件利润率、软件订阅、虚拟商品三者如何组合？
- 第一代产品应该先卖硬件，还是先做小程序社区和虚拟形象？
### 6. 目标用户验证问题
- 第一批种子用户应该是谁？
应该通过小红书、TikTok、B站、Instagram、独立站还是众筹平台验证？
- 用户购买理由是什么？
- 用户复购或持续付费理由是什么？
哪些人会觉得这个产品“幼稚”，哪些人会觉得“有趣、有陪伴感”？
### 7. 风险问题
- 是否容易被认为只是普通毛绒玩具加震动？
- 是否容易被 AI 玩具赛道同质化竞争淹没？
- 硬件复杂度是否会导致成本过高？
- 发热与电池安全是否会增加认证难度？
- 软件端故事流是否足够形成长期留存？
- 海外成人市场的产品边界和合规边界如何把握？
